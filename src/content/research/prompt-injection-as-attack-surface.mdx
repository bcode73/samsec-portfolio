For most of the last two years, prompt injection has been treated as a curiosity: a way to make a chatbot say
something embarrassing. That framing is out of date. As large language models move from chat interfaces into
autonomous agents with access to tools, file systems, browsers, and APIs, the same vulnerability class becomes
something closer to remote code execution for natural language.

This piece explains why prompt injection deserves to be treated as a first-class attack surface, how it differs
structurally from injection vulnerabilities security teams already understand, and what a defensible
architecture for agentic systems looks like.

## Why this is not "just" SQL injection with words

SQL injection has a well-understood fix: parameterize the query, separate code from data, and the class of bug
mostly disappears. Prompt injection resists this fix because, in most current LLM architectures, there is no
hard boundary between instructions and data. Both arrive as tokens in the same context window, interpreted by
the same model, with the same authority.

<Callout type="note">
An agent that reads untrusted content (a webpage, an email, a PDF, a support ticket) and can also take actions
is, by construction, executing instructions from that untrusted content unless something actively prevents it.
</Callout>

That is the core problem: **any content an agent ingests is a potential instruction channel**, whether or not
the system designer intended it to be.

## The expanding blast radius

The severity of prompt injection scales directly with what the agent is permitted to do. A chatbot that can
only answer questions has a narrow blast radius. Worst case, it says something wrong or offensive. An agent
with:

- Browser or web-fetch access
- File system read and write access
- Email or messaging send capability
- API credentials to internal systems
- Code execution

has a blast radius closer to a compromised service account. Security teams evaluating AI agent deployments
should treat each additional tool grant as an expansion of attack surface, not a convenience feature to be
added by default.

## Where injected instructions hide

In practice, injected instructions rarely arrive as an obvious "ignore previous instructions" string. That
pattern is easy to filter. The more effective payloads are the ones that don't look like attacks:

1. **Indirect injection via retrieved content.** A webpage, document, or ticket the agent is asked to
   summarize, containing hidden or benign-looking instructions.
2. **Metadata and structured fields.** Filenames, HTTP headers, commit messages, or alt text that a naive
   pipeline feeds directly into the model's context.
3. **Multi-turn erosion.** A sequence of individually reasonable requests that, combined, push the agent past
   a boundary a single request would have triggered a refusal for.
4. **Tool output poisoning.** A compromised or malicious upstream API returning a response engineered to be
   reinterpreted as an instruction once it re-enters the model's context.

## Toward a defensible architecture

There is no complete fix available today. But there is a real difference between systems built with this
threat model in mind and systems that bolt AI onto existing infrastructure without reconsidering trust
boundaries. A few principles we apply when designing agentic workflows inside SamSec Ops:

### Separate privilege from content

The model's context window should never be the sole authority for what actions are permitted. Tool access
should be gated by a policy layer outside the model's control, one that can't be argued with, tricked, or
socially engineered, because it isn't a language model.

### Treat every external input as untrusted, including tool output

This includes search results, scraped pages, API responses, and file contents, not just direct user input.
Once something enters the context window, it should be handled with the same suspicion regardless of its
source.

### Constrain action space before constraining prompts

Prompt-level defenses (instructions telling the model to ignore attempts to override its instructions) are
useful as a layer, but they are not a security boundary. They are a heuristic that degrades under adversarial
pressure. The real boundary is what actions the system will physically permit, independent of what the model
decides to do.

### Make consequential actions reviewable

For actions with real-world effect, such as sending communications, modifying data, or executing code, a
human-in-the-loop checkpoint or a strict allowlist of pre-approved action patterns closes most of the
remaining gap.

```text
Untrusted input → Model reasoning → Proposed action
                                          │
                                          ▼
                          Policy layer (outside model control)
                                          │
                              allowed ────┴──── denied / review
```

## What this means for security teams

Prompt injection is not a reason to avoid building AI agents into security or engineering workflows. It's a
reason to build them deliberately. The organizations that get the most value out of agentic AI over the next
several years will be the ones treating this as a design constraint from day one, not the ones racing to grant
the broadest tool access and hoping the model behaves.

This is also why [SamSec Agent](/samsec-ops#agent) keeps AI behind an explicit policy boundary between what
an agent can *reason about* and what it can *do*: risk classification, explicit confirmation for destructive
actions, and an authorization attestation before any engagement begins. That distinction is easy to blur
under product pressure, and expensive to retrofit once it has been. The [five principles that guide agent
design at
SamSec](/notes/five-principles-secure-ai-agent-design) go into the practical detail behind that boundary, which
is formalized as [ADR-002: Human Authorization Before Autonomous Execution](/adr/adr-002-human-authorization)
and examined at length in [Human Oversight in Autonomous
Infrastructure](/research/human-oversight-in-autonomous-infrastructure).
